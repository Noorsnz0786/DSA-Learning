Section 1 · Foundations

Thinking & Speed

Before any data structure: a way to attack problems, and the language for "how fast is it?" — exactly what a verbal interviewer is testing.

A real-life way to attack any problem

Real life A friend asks you to find one book in a messy room. You don't dive in blindly — you ask "what does it look like?" (understand), then "I'll check every shelf" (a first plan), then "wait, it's alphabetised — jump to the M's" (a better plan). That out-loud reasoning is the interview.

Say this 4-step track out loud on every problem — call it UPER:

StepWhat you literally say
U — Understand"Input is ___, output is ___. Can I assume ___? Here's a tiny example."
P — Plan brute force"Simplest thing that works: try every possibility. That's probably O(n²)…"
E — Enhance"Can I do better? If I store seen values in a hash map, I drop the inner loop → O(n)."
R — Run itCode it, then trace your tiny example by hand to check it.
Why brute force first Saying the slow solution out loud first guarantees you always have an answer, and sets up the "can I do better?" moment where you shine. Silence while thinking looks worse than a spoken brute force.

Big-O — how we measure "fast"

"Fast" can't mean seconds (depends on the computer). So we measure how the number of steps grows as the input grows. We call the input size n.

Real life #1 — the dictionary Reading every page to find a word grows with the book — twice the pages, twice the work (O(n)). Opening to the middle and halving each time barely grows — a dictionary 1000× bigger needs only ~10 more steps (O(log n)).
Real life #2 — handshakes at a party If everyone shakes hands with everyone, handshakes grow like O(n²). A few more guests, and they balloon. That's why nested loops over the same data get expensive fast.
Big-ONameEveryday meaning
O(1)ConstantSame effort regardless of size. Grab the first item.
O(log n)LogarithmicHalve the problem each step. Dictionary lookup.
O(n)LinearLook at each item once. Read every page.
O(n log n)LinearithmicThe speed of good sorting.
O(n²)QuadraticFor each item, look at every item. Party handshakes.

Space complexity (the quick sibling)

Same idea, for extra memory. A new list holding all n items = O(n) space. A couple of variables = O(1) space. They often ask for both.

Real life Counting people in a room: a tally in your head is O(1) memory; writing every name on a list is O(n) memory. Same answer, different memory cost.
Say it in the interview Drill this: "My brute force is O(n²) time, O(1) space. I think I can trade memory for speed — a hash map should get it to O(n) time, O(n) space." Say it out loud now.

1 · Two Sum  ＋easy

Return indices of the two numbers that add to target. [2,7,11,15], 9 → [0,1].

Approach 1 — Brute force (every pair)
def two_sum(nums, target):
    # Try every pair (i, j) with j after i
    for i in range(len(nums)):
        for j in range(i + 1, len(nums)):   # j starts after i so we don't reuse one element
            if nums[i] + nums[j] == target:
                return [i, j]
    return []                                # no pair found
# Time O(n²)  ·  Space O(1)
Approach 2 — Hash map, one pass best
def two_sum(nums, target):
    seen = {}                       # maps value -> its index
    for i, x in enumerate(nums):    # walk through once
        need = target - x           # the partner x needs to hit target
        if need in seen:            # O(1): have we already passed that partner?
            return [seen[need], i]  # yes -> we have our pair
        seen[x] = i                 # otherwise remember x for the future
    return []
# Time O(n)  ·  Space O(n)

Why best: the hash map removes the inner loop. One pass, O(1) lookups → O(n). The classic memory-for-speed trade.

Show the hash-map version stepping through
Talk-track"Brute force is every pair, O(n²). I'll store seen numbers in a hash map and check if each number's partner (target − x) was already seen — O(1) per check, one pass, O(n) time and O(n) space."
2 · Contains Duplicate  ＋easy

Return True if any value appears twice. [1,2,3,1] → True.

Approach 1 — Brute force
def contains_duplicate(nums):
    for i in range(len(nums)):
        for j in range(i + 1, len(nums)):
            if nums[i] == nums[j]:   # compare every pair
                return True
    return False
# Time O(n²)  ·  Space O(1)
Approach 2 — Sort first, then check neighbours
def contains_duplicate(nums):
    nums.sort()                       # duplicates become adjacent
    for i in range(1, len(nums)):
        if nums[i] == nums[i - 1]:    # compare each item to the one before
            return True
    return False
# Time O(n log n)  ·  Space O(1)
Approach 3 — Hash set best
def contains_duplicate(nums):
    seen = set()                # a set = hash map with keys only
    for x in nums:
        if x in seen:           # O(1): seen this value before?
            return True
        seen.add(x)             # remember it
    return False
# Time O(n)  ·  Space O(n)

Why best: one pass, O(1) membership checks. Mention the sort version too — it's O(1) space if they forbid extra memory.

Talk-track"A set remembers what I've seen and checks membership in O(1), so one pass is O(n) time, O(n) space. If they want O(1) space, I'd sort and compare neighbours — O(n log n)."
3 · Valid Anagram  ＋easy

Is t a rearrangement of s? "anagram","nagaram" → True.

Approach 1 — Sort both
def is_anagram(s, t):
    return sorted(s) == sorted(t)   # same letters -> same sorted order
# Time O(n log n)  ·  Space O(n)
Approach 2 — Count letters best
from collections import Counter
def is_anagram(s, t):
    if len(s) != len(t):       # quick reject: different lengths can't match
        return False
    return Counter(s) == Counter(t)   # compare letter -> count maps
# Time O(n)  ·  Space O(1)  (at most 26 letters)

Why best: counting is O(n) vs sorting's O(n log n). Counter builds a hash map of letter→count.

4 · Best Time to Buy & Sell Stock  ＋easy

One buy, one later sell. Max profit? [7,1,5,3,6,4] → 5 (buy 1, sell 6).

Approach 1 — Brute force (every buy/sell pair)
def max_profit(prices):
    best = 0
    for i in range(len(prices)):
        for j in range(i + 1, len(prices)):    # sell day after buy day
            best = max(best, prices[j] - prices[i])
    return best
# Time O(n²)  ·  Space O(1)
Approach 2 — Track cheapest so far, one pass best
def max_profit(prices):
    min_price = float('inf')        # cheapest day seen so far
    profit = 0                      # best profit so far
    for p in prices:
        if p < min_price:           # new cheapest buy day
            min_price = p
        elif p - min_price > profit: # selling today beats our record?
            profit = p - min_price
    return profit
# Time O(n)  ·  Space O(1)

Why best: we only ever need the lowest price seen so far — no need to recheck old days. One pass, no extra memory.

Talk-track"I track the cheapest price so far and, at each day, ask 'if I sold today, is that my best profit?' One pass, O(n) time, O(1) space."
5 · Maximum Subarray (Kadane's)  ＋medium

Largest sum of a contiguous run. [-2,1,-3,4,-1,2,1,-5,4] → 6 (the run [4,-1,2,1]).

Approach 1 — Brute force (every subarray)
def max_subarray(nums):
    best = nums[0]
    for i in range(len(nums)):
        total = 0
        for j in range(i, len(nums)):   # every run starting at i
            total += nums[j]
            best = max(best, total)
    return best
# Time O(n²)  ·  Space O(1)
Approach 2 — Kadane's algorithm best
def max_subarray(nums):
    best = nums[0]          # best sum found anywhere
    current = nums[0]       # best sum of a run ENDING at this index
    for x in nums[1:]:
        # either extend the current run, or start fresh at x
        current = max(x, current + x)
        best = max(best, current)
    return best
# Time O(n)  ·  Space O(1)

Why best: the key insight — if the running sum goes negative it can only hurt, so drop it and start fresh. One pass.

▶ Watch Kadane's run step by step
Talk-track"At each element I decide: extend the current run or restart from here. I keep the best seen. A negative running sum is worth abandoning. O(n) time, O(1) space."
6 · Group Anagrams  ＋medium

Group words that are anagrams. ["eat","tea","tan","ate"] → [["eat","tea","ate"],["tan"]].

Approach — sorted word as a hash-map key best
from collections import defaultdict
def group_anagrams(words):
    groups = defaultdict(list)        # key -> list of words
    for w in words:
        key = ''.join(sorted(w))      # "eat" and "tea" both sort to "aet"
        groups[key].append(w)         # same key -> same group
    return list(groups.values())
# Time O(n · k log k)  ·  Space O(n·k)   (k = word length)

Key idea: anagrams share the same sorted letters, so use that as a dictionary key to bucket them.

Talk-track"Anagrams sort to the same string, so I use the sorted word as a hash-map key and append each word to its bucket. One pass over n words, each costing a sort of length k."
7 · Majority Element  ＋easy

Element appearing more than n/2 times (guaranteed to exist). [2,2,1,1,1,2,2] → 2.

Approach 1 — Count with a hash map
from collections import Counter
def majority(nums):
    counts = Counter(nums)                 # value -> how many times
    return max(counts, key=counts.get)     # the most frequent key
# Time O(n)  ·  Space O(n)
Approach 2 — Boyer–Moore voting best
def majority(nums):
    count = 0
    candidate = None
    for x in nums:
        if count == 0:        # no leader -> adopt x as candidate
            candidate = x
        count += 1 if x == candidate else -1   # vote up / down
    return candidate
# Time O(n)  ·  Space O(1)

Why best: O(1) space. The majority element survives the cancelling because it outnumbers everything else combined.

▶ Watch Boyer–Moore voting run step by step
Talk-track"Simplest: count frequencies in a hash map — O(n)/O(n). To use O(1) space, Boyer–Moore voting: matching votes cancel opposing ones, so the true majority is left standing."

Section 3 · Technique

Two Pointers

One of the highest-value tricks: use two markers instead of a nested loop, and turn many O(n²) problems into O(n). The 7 most-asked two-pointer problems — every approach, commented, traceable.

The idea in one picture

Real life Two friends search a long bookshelf for a gap to fit a new book — one starts at the left end, one at the right end, and they walk toward each other. They cover the whole shelf in one sweep instead of each checking every spot. Two pointers = two markers moving through the data with a rule for which one moves next.

There are two common flavours:

FlavourHow the pointers moveUsed for
ConvergingOne at each end, moving inward until they meet.Palindrome, two-sum on a sorted array, container with water, reversing.
Fast / slowBoth start left; one races ahead, the other lags.Removing duplicates, moving zeroes, in-place edits.
When to reach for it Think two pointers when the array is sorted, or when you're comparing/​editing from both ends, or need an in-place O(1)-space result. The giveaway: a brute force that pairs every element with every other (O(n²)) on data where order helps.

1 · Valid Palindrome  ＋easy

Is the string the same forwards and backwards, ignoring case and non-letters? "A man, a plan, a canal: Panama" → True.

Approach 1 — Clean, then compare to its reverse
def is_palindrome(s):
    cleaned = [c.lower() for c in s if c.isalnum()]  # keep only letters/digits, lowercased
    return cleaned == cleaned[::-1]                  # [::-1] reverses the list
# Time O(n)  ·  Space O(n)   (builds a reversed copy)
Approach 2 — Two pointers from the ends best
def is_palindrome(s):
    left, right = 0, len(s) - 1
    while left < right:
        while left < right and not s[left].isalnum():  # skip junk from the left
            left += 1
        while left < right and not s[right].isalnum(): # skip junk from the right
            right -= 1
        if s[left].lower() != s[right].lower():        # mismatch -> not a palindrome
            return False
        left += 1
        right -= 1
    return True
# Time O(n)  ·  Space O(1)   (no extra copy)

Why best: O(1) space — we never build a second string, just walk inward.

Talk-track"I put one pointer at each end and walk inward, comparing as I go and skipping non-alphanumerics. Any mismatch means not a palindrome. O(n) time, O(1) space — no reversed copy."
2 · Two Sum II — Sorted Input  ＋medium

The array is sorted. Return the 1-indexed positions of the two numbers adding to target. [2,7,11,15], 9 → [1,2].

Approach — Converging pointers best
def two_sum_sorted(nums, target):
    left, right = 0, len(nums) - 1
    while left < right:
        s = nums[left] + nums[right]
        if s == target:
            return [left + 1, right + 1]   # problem wants 1-indexed
        elif s < target:
            left += 1                       # sum too small -> raise the low end
        else:
            right -= 1                      # sum too big -> lower the high end
# Time O(n)  ·  Space O(1)

Why it works: because it's sorted, moving left right increases the sum and moving right left decreases it — so we always know which pointer to move. No hash map needed.

▶ Watch it run step by step
Talk-track"Since it's sorted, I start wide and squeeze: if the sum's too big I move the right pointer down, too small I move the left up. O(n) time, O(1) space — better than the hash-map version's O(n) space."
3 · Container With Most Water  ＋medium

Each number is a wall height. Pick two walls holding the most water. [1,8,6,2,5,4,8,3,7] → 49.

Approach 1 — Brute force (every pair)
def max_area(height):
    best = 0
    for i in range(len(height)):
        for j in range(i + 1, len(height)):
            area = min(height[i], height[j]) * (j - i)   # shorter wall × width
            best = max(best, area)
    return best
# Time O(n²)  ·  Space O(1)
Approach 2 — Two pointers, move the shorter wall best
def max_area(height):
    left, right = 0, len(height) - 1
    best = 0
    while left < right:
        h = min(height[left], height[right])
        best = max(best, h * (right - left))   # area with current walls
        if height[left] < height[right]:
            left += 1                # the short wall limits us — try to replace it
        else:
            right -= 1
    return best
# Time O(n)  ·  Space O(1)

Key insight: the shorter wall caps the area, so moving the taller one can only shrink the width without raising the limit — always move the shorter wall.

Talk-track"Brute force checks every pair — O(n²). With two pointers I start at the widest gap and always move the shorter wall inward, since it's the limiting factor. One pass, O(n) time, O(1) space."
4 · Remove Duplicates from Sorted Array  ＋easy

Sorted array — remove duplicates in place, return the new length. [1,1,2,3,3] → 3 (array becomes [1,2,3,...]).

Approach — Fast / slow pointers best
def remove_duplicates(nums):
    if not nums:
        return 0
    slow = 0                            # last index of the unique part
    for fast in range(1, len(nums)):    # fast scans ahead
        if nums[fast] != nums[slow]:    # found a new value
            slow += 1
            nums[slow] = nums[fast]     # place it right after the unique part
    return slow + 1                     # count = last index + 1
# Time O(n)  ·  Space O(1)

Idea: slow marks the end of the "clean" unique section; fast looks for the next new value to append there.

▶ Watch the fast/slow pointers run step by step
Talk-track"Two pointers: slow holds the end of the unique prefix, fast scans for the next different value and copies it just past slow. Since it's sorted, duplicates are adjacent. O(n) time, O(1) space."
5 · Move Zeroes  ＋easy

Move all 0s to the end, keep the order of the rest, in place. [0,1,0,3,12] → [1,3,12,0,0].

Approach — Fast / slow with swap best
def move_zeroes(nums):
    slow = 0                       # next spot for a non-zero value
    for fast in range(len(nums)):
        if nums[fast] != 0:
            nums[slow], nums[fast] = nums[fast], nums[slow]  # swap non-zero forward
            slow += 1
    return nums
# Time O(n)  ·  Space O(1)

Idea: slow always points at where the next non-zero should land. Every non-zero gets swapped forward; zeroes naturally bubble to the back.

▶ Watch it run step by step
Talk-track"slow tracks where the next non-zero belongs. As fast finds non-zeros, I swap them into slow's spot. Order is preserved and zeroes end up at the back. O(n) time, O(1) space."
6 · Reverse String (in place)  ＋easy

Reverse a list of characters in place. ['h','e','l','l','o'] → ['o','l','l','e','h'].

Approach — Converging pointers, swap best
def reverse_string(s):
    left, right = 0, len(s) - 1
    while left < right:
        s[left], s[right] = s[right], s[left]   # swap the two ends
        left += 1
        right -= 1
    return s
# Time O(n)  ·  Space O(1)

Idea: swap the outermost pair, then move both pointers inward. They meet in the middle.

▶ Watch it run step by step
Talk-track"Swap the ends and walk inward until the pointers meet. In place, so O(1) extra space, O(n) time."
7 · 3Sum (the famous one)  ＋medium

Find all unique triplets that sum to 0. [-1,0,1,2,-1,-4] → [[-1,-1,2],[-1,0,1]].

Approach — Sort, fix one, two pointers on the rest best
def three_sum(nums):
    nums.sort()                      # sorting unlocks the two-pointer move
    res = []
    for i in range(len(nums)):
        if i > 0 and nums[i] == nums[i-1]:
            continue                 # skip a repeated anchor to avoid dup triplets
        left, right = i + 1, len(nums) - 1
        while left < right:
            total = nums[i] + nums[left] + nums[right]
            if total == 0:
                res.append([nums[i], nums[left], nums[right]])
                left += 1
                right -= 1
            elif total < 0:
                left += 1            # need a bigger sum
            else:
                right -= 1           # need a smaller sum
    return res
# Time O(n²)  ·  Space O(1) (ignoring output)

Key idea: 3Sum = "fix one number, then it's just Two Sum II on the rest". Sorting makes the inner two-pointer sweep work and makes duplicates easy to skip. Brute force is three nested loops, O(n³); this is O(n²).

Talk-track"I sort, then fix each number as an anchor and run a two-pointer sweep on the rest looking for the negative of the anchor — that's Two Sum on a sorted array. Skipping duplicate anchors keeps triplets unique. O(n²) time, way better than the O(n³) brute force."

Sliding Window

When a problem asks about a contiguous chunk — a subarray or substring — don't recompute from scratch each time. Slide a window: add what enters, drop what leaves. Turns O(n·k) and O(n²) into O(n). The 6 most-asked window problems, every approach, commented, traceable.

The idea in one picture

Real life Looking out of a moving train, you see 3 seats at a time. As the train rolls one seat forward, one new seat enters your view and one leaves — you don't re-count all three, you just adjust by the difference. A "window" is that frame sliding across your data.
Why it's fast Naive: for every starting point, re-add all k elements → O(n·k). Sliding window: reuse the previous sum, just + entering − leaving → O(1) per move, O(n) total. You stop throwing away work you already did.

Two flavours — know which one a problem wants:

FlavourWindow sizeSignal in the problem
FixedAlways exactly k."…of size k", "every k consecutive…". Slide by one each step.
VariableGrows and shrinks."longest/shortest subarray such that…". Expand right; shrink left while a condition holds.
The variable-window recipe Almost every variable-window solution is the same shape: expand right by one each loop; while the window breaks the rule, shrink left; then record the best answer. Learn this skeleton and half these problems write themselves.

from collections import Counter
def find_anagrams(s, p):
    need = Counter(p)                # letters p needs, e.g. {a:1, b:1}
    k = len(p)
    window = Counter(s[:k])          # first window's letter counts
    res = []
    if window == need:
        res.append(0)
    for i in range(k, len(s)):       # slide the fixed window
        window[s[i]] += 1                # add entering letter
        window[s[i-k]] -= 1              # remove leaving letter
        if window[s[i-k]] == 0:
            del window[s[i-k]]           # keep map clean for == to work
        if window == need:
            res.append(i - k + 1)
    return res
# Time O(n)  ·  Space O(1)  (≤26 letters)

Idea: two strings are anagrams iff their letter counts match. Keep a fixed window of counts and slide it, comparing to need each step.

def char_replacement(s, k):
    count = {}
    left = 0
    best = 0
    max_freq = 0                     # count of the most common char in window
    for right in range(len(s)):

            left += 1

    return best

Stacks & Queues

Two tiny structures with huge interview payoff. A stack remembers the "most recent" thing; a queue remembers the "oldest". Master push/pop and the monotonic-stack trick. The 6 most-asked problems — every approach, commented, traceable.

Stack — last in, first out (LIFO)

Real life A stack of plates. You add to the top and take from the top — the last plate on is the first one off. Same as your browser's Back button or Ctrl-Z undo: it returns the most recent thing first. 
stack = []
stack.append(10)   # push — add to top, O(1)
stack.append(20)
stack[-1]          # peek the top → 20, O(1)
stack.pop()        # remove the top → 20, O(1)

Queue — first in, first out (FIFO)

Real life A line at a ticket counter. First person in line is served first. Print jobs, task queues, and (later) breadth-first search all work this way. 
from collections import deque
q = deque()
q.append(10)       # enqueue — add to back, O(1)
q.append(20)
q.popleft()        # dequeue — remove from front → 10, O(1)
Python gotcha Use collections.deque for a queue, NOT a list. list.pop(0) removes from the front in O(n) (everything shifts). deque.popleft() is O(1). A list is fine as a stack (append/pop from the end are O(1)).
When to reach for a stack Matching/nesting (brackets, tags), anything "undo" or "most recent", evaluating expressions, and — the big interview one — "next greater/smaller element" problems, solved with a monotonic stack (a stack kept in increasing or decreasing order). If you ever think "for each item, find the nearest bigger/smaller one", reach for a monotonic stack to turn O(n²) into O(n).

1 · Valid Parentheses (the classic)  ＋easy

2 · Min Stack  ＋medium

Design a stack with push, pop, top, and getMin — all in O(1).

class MinStack:
    def __init__(self):
        self.stack = []                      # each entry: (value, min at/below it)
    def push(self, x):
        m = min(x, self.stack[-1][1]) if self.stack else x
        self.stack.append((x, m))            # remember the running min with each value
    def pop(self):
        self.stack.pop()
    def top(self):
        return self.stack[-1][0]
    def getMin(self):
        return self.stack[-1][1]             # min is right there — O(1)
# Every operation O(1)  ·  Space O(n)

Key trick: a naive min() scan is O(n). Instead, store the minimum-so-far alongside each value. When you pop, the previous min is automatically exposed again.

4 · Daily Temperatures (monotonic stack)  ＋medium

For each day, how many days until a warmer one? [73,74,72,76] → [1,2,1,0].

def daily_temperatures(temps):

    for i in range(len(temps)):
        for j in range(i + 1, len(temps)):    # scan ahead for a warmer day
            if temps[j] > temps[i]:
                res[i] = j - i
                break
    return res
# Time O(n²)  ·  Space O(1)
Approach 2 — Monotonic stack of indices best
def daily_temperatures(temps):
    res = [0] * len(temps)
    stack = []                                # indices of days still waiting for a warmer day
    for i, t in enumerate(temps):
        while stack and temps[stack[-1]] < t:  # today is warmer than waiting days
            j = stack.pop()
            res[j] = i - j                     # days that day waited
        stack.append(i)
    return res
# Time O(n)  ·  Space O(n)

Key idea: keep a stack of days whose warmer day hasn't arrived. When a warmer day comes, it resolves all the cooler days waiting on top. Each index is pushed and popped once → O(n).

5 · Implement Queue using Stacks  ＋easy

Approach — Two stacks, refill on demand best
class MyQueue:
    def __init__(self):
        self.inbox = []                  # new items pushed here
        self.outbox = []                 # items leave from here
    def push(self, x):
        self.inbox.append(x)
    def pop(self):
        self._shift()
        return self.outbox.pop()
    def peek(self):
        self._shift()
        return self.outbox[-1]
    def _shift(self):
        if not self.outbox:              # only refill when outbox is empty
            while self.inbox:
                self.outbox.append(self.inbox.pop())   # reversing restores FIFO order
# Amortized O(1) per op  ·  Space O(n)

The trick: a stack reverses order. Pour inbox into outbox and it reverses again — back to the original (FIFO) order. Each item moves at most once, so it averages O(1).

1 · Reverse Linked List (the classic)  ＋easy

def reverse_list(head):

    curr = head

        nxt = curr.next          # SAVE next before we overwrite the link

        curr = nxt               # advance curr

Section 7 · Technique

Recursion

A function that solves a problem by calling itself on a smaller version of the same problem. This is the engine behind trees, graphs, divide-and-conquer, and dynamic programming — so it's worth getting deep in your bones.

The idea in one picture

Real life Russian nesting dolls. To count them: open the doll — inside is a smaller doll (the same problem, smaller). Keep opening until you reach the tiny solid one that doesn't open (the base case). Then close them back up, counting as you go.

Every recursive function has exactly two parts. Get these right and recursion is easy:

PartWhat it does
Base caseThe smallest input you can answer directly, with no further recursion. The "stop" condition.
Recursive caseCall yourself on a smaller input, then combine that answer with the current step.
def factorial(n):
    if n == 1:           # BASE CASE — stop here
        return 1
    return n * factorial(n - 1)   # RECURSIVE CASE — smaller problem
The #1 bug Forgetting the base case, or recursing on something that isn't actually smaller → infinite recursion → stack overflow (Python: "maximum recursion depth exceeded"). Always ask: "what's my stop condition, and does each call get closer to it?"
The call stack — what's really happening Each call pauses and waits for its sub-call to finish, stacking up "unfinished work" (this is literally a stack — Section 5!). When the base case returns, the calls resolve one by one on the way back up. The animation below makes this visible. Understanding it is the key to reading trees and graphs later.

        return 1
    return n * factorial(n - 1)  # n! = n × (n-1)!
# Time O(n)  ·  Space O(n)  (n stacked calls)

Read it aloud: "factorial of n is n times factorial of n−1, and factorial of 1 is just 1." That sentence is the code. Note: space is O(n) because n calls stack up before any returns.

    if n % 2 == 0:

    else:

# Time O(log n)  ·  Space O(log n)

4 · Fibonacci (and why naive recursion is slow)  ＋medium

Each number is the sum of the previous two. 0,1,1,2,3,5,8…  fib(6) → 8.

Approach 1 — Naive recursion (correct but slow!)
def fib(n):
    if n < 2:                    # base: fib(0)=0, fib(1)=1
        return n
    return fib(n - 1) + fib(n - 2)
# Time O(2ⁿ)  ·  Space O(n)   — recomputes the same values over and over
Approach 2 — Memoization (cache results) best
def fib(n, memo={}):
    if n < 2:
        return n
    if n in memo:                # already computed? reuse it
        return memo[n]
    memo[n] = fib(n-1, memo) + fib(n-2, memo)   # store before returning
    return memo[n]
# Time O(n)  ·  Space O(n)

The big lesson (bridge to DP!): naive Fibonacci recomputes fib(2), fib(3)… a huge number of times — that's the O(2ⁿ) explosion. Memoization caches each answer the first time, so it's computed once → O(n). This caching idea is the heart of dynamic programming, coming up later.

6 · Generate All Subsets (intro to backtracking)  ＋medium

Return every subset (the "power set"). [1,2] → [[],[1],[1,2],[2]].

Approach — Choose / explore / un-choose (backtracking) best
def subsets(nums):
    res = []
    def backtrack(start, path):
        res.append(path[:])          # every node is a valid subset — record a copy
        for i in range(start, len(nums)):
            path.append(nums[i])     # CHOOSE nums[i]
            backtrack(i + 1, path)   # EXPLORE with it included
            path.pop()               # UN-CHOOSE (backtrack) to try the next
    backtrack(0, [])
    return res
# Time O(n · 2ⁿ)  ·  Space O(n) for the recursion path

The backtracking template: choose → recurse → un-choose. This same skeleton generates permutations, combinations, and solves puzzles like N-Queens. path[:] records a copy (since path keeps changing).

Binary Search

Halve the search space every step → O(log n). Trivial to describe, easy to get wrong on the boundaries. Nail one template, then learn the "search on the answer" idea that unlocks a whole class of problems. The 6 most-asked, every approach, commented, traceable.

The idea (you've met it before)

Real life Guess-the-number: "I'm thinking of a number 1–100." You say 50 — "higher". You say 75 — "lower". Each guess halves what's left. You corner any number in ~7 guesses. That halving is binary search, and it's why it's O(log n).
The one precondition The data must be sorted — or more generally, there's a point where a yes/no condition flips from false to true (it's monotonic). Without that, halving tells you nothing. So a great clarifying question is: "is the input sorted?"

Two templates cover almost everything. Mixing them up is the classic bug:

GoalLoopUpdates
Find an exact valuewhile lo <= hilo = mid+1 / hi = mid-1, return on match
Find a boundary (first/last that satisfies X)while lo < hihi = mid / lo = mid+1, return lo
The pitfalls that fail interviews (1) Infinite loop — the search space must shrink every iteration; with while lo < hi and hi = mid, make sure the other branch is lo = mid+1. (2) Off-by-one on mid±1. (3) Forgetting it must be sorted. Say your invariant out loud: "the answer is always within [lo, hi]."

1 · Binary Search (the template)  ＋easy

Return the index of target in a sorted array, or −1. [-1,0,3,5,9,12], 9 → 4.

Approach — Exact-match template best
def binary_search(nums, target):
    lo, hi = 0, len(nums) - 1
    while lo <= hi:                  # search space is [lo, hi]
        mid = (lo + hi) // 2
        if nums[mid] == target:
            return mid
        elif nums[mid] < target:
            lo = mid + 1             # target is in the right half
        else:
            hi = mid - 1             # target is in the left half
    return -1                        # not found
# Time O(log n)  ·  Space O(1)

Memorise this template. Every other binary search is a variation. The invariant: if the target exists, it's always inside [lo, hi].

Talk-track"It's sorted, so I check the middle: equal means found; smaller means search right; bigger means search left. The space halves each step → O(log n) time, O(1) space."

4 · Search in Rotated Sorted Array (the famous one)  ＋medium

A sorted array rotated at some pivot. Find target in O(log n). [4,5,6,7,0,1,2], 0 → 4.

Approach — One half is always sorted; binary-search that best
def search_rotated(nums, target):
    lo, hi = 0, len(nums) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if nums[mid] == target:
            return mid
        if nums[lo] <= nums[mid]:            # left half [lo..mid] is sorted
            if nums[lo] <= target < nums[mid]:
                hi = mid - 1                 # target is in that sorted half
            else:
                lo = mid + 1
        else:                                 # right half [mid..hi] is sorted
            if nums[mid] < target <= nums[hi]:
                lo = mid + 1
            else:
                hi = mid - 1
    return -1
# Time O(log n)  ·  Space O(1)

Key insight: even after rotation, at least one side of mid is fully sorted. Figure out which, check if the target lies inside that sorted side, and discard half — staying O(log n).

6 · Sqrt(x) — binary search on the answer  ＋easy

Integer square root (floor). sqrt(8) → 2, sqrt(16) → 4.

def my_sqrt(x):

    ans = 0
    while lo <= hi:
        mid = (lo + hi) // 2
        if mid * mid <= x:
            ans = mid                # mid works — remember it, try bigger
            lo = mid + 1
        else:
            hi = mid - 1             # mid² too big — go smaller
    return ans
# Time O(log x)  ·  Space O(1)

The big idea — "binary search on the answer": there's no array here. We binary-search the range of possible answers (0 to x), using a monotonic check (mid² ≤ x). This same pattern solves Koko Eating Bananas, ship-capacity, and "minimum X such that…" problems. Spotting it is a senior-level move.

Trees & Binary Search Trees

Hierarchical data — each node points to up to two children. This is where recursion (Section 7) pays off hugely, and where BFS uses the queue (Section 5). The 6 most-asked problems, every approach, commented, traceable.

What a tree is

Real life A family tree, an org chart, or folders on your computer. One thing at the top (the root), branching down into children, children of children… ending in leaves (nodes with no children). No loops — every node has exactly one parent. 
class TreeNode:
    def __init__(self, val=0, left=None, right=None):
        self.val = val
        self.left = left      # left child (a TreeNode or None)
        self.right = right    # right child

Quick vocabulary: root (top), leaf (no children), height/depth (longest root-to-leaf path), subtree (a node + everything below it).

Binary Search Tree (BST) — the ordered tree

The BST property For every node: everything in the left subtree is smaller, everything in the right subtree is larger. That ordering lets you search, insert, and delete in O(h) — O(log n) if the tree is balanced — by going left or right like binary search. Bonus fact you'll use: an inorder traversal of a BST visits values in sorted order.

Traversals — the 4 ways to visit every node

TraversalOrderTool / use
Preorder (DFS)node → left → rightRecursion. Copy/serialize a tree.
Inorder (DFS)left → node → rightRecursion. BST → sorted order.
Postorder (DFS)left → right → nodeRecursion. Delete/compute from leaves up.
Level-order (BFS)top to bottom, left to rightA queue. Shortest path, level grouping.
The mental model DFS = "go deep down one branch first" — naturally recursive (the call stack does the work). BFS = "explore level by level" — uses a queue. Almost every tree problem is one of these two with a tiny twist.

1 · Maximum Depth of Binary Tree  ＋easy

How many levels deep is the tree? [3,9,20,null,null,15,7] → 3.

def max_depth(root):

    right = max_depth(root.right)    # depth of right subtree

4 · Binary Tree Level Order Traversal (BFS)  ＋medium

Return values level by level. [3,9,20,null,null,15,7] → [[3],[9,20],[15,7]].

Approach — BFS with a queue best
from collections import deque
def level_order(root):
    if not root:
        return []
    result = []
    queue = deque([root])
    while queue:
        level = []
        for _ in range(len(queue)):      # freeze the count = exactly this level
            node = queue.popleft()
            level.append(node.val)
            if node.left:  queue.append(node.left)    # enqueue next level
            if node.right: queue.append(node.right)
        result.append(level)
    return result
# Time O(n)  ·  Space O(n)

The key trick: capture len(queue) before the inner loop — that's exactly how many nodes are on the current level, so each pass collects one full level. This is the BFS template for trees.

▶ Watch the queue process level by level
Talk-track"BFS with a queue. I record the queue's length to know how many nodes are on the current level, pop exactly that many, collect their values, and enqueue their children for the next level. O(n) time and space."

def is_valid_bst(root, lo=float('-inf'), hi=float('inf')):

    return (is_valid_bst(root.left, lo, root.val) and   # left: tighten the upper bound

def lca(root, p, q):

Graphs (BFS & DFS)

Nodes connected by edges — the most general structure. A tree is just a graph with no cycles. Master two traversals (BFS & DFS) and a visited set, and a huge class of "regions / reachability / ordering" problems opens up. The 6 most-asked, every approach, commented, traceable.

What a graph is

Real life A social network: people are nodes, friendships are edges. Or a map: cities and roads. Or course prerequisites. Unlike a tree, a graph can have cycles (A→B→C→A) and a node can have many "parents".

Almost always represented as an adjacency list — a dict mapping each node to its neighbours:

graph = {
    0: [1, 2],     # node 0 connects to 1 and 2
    1: [0, 3],
    2: [0, 3],
    3: [1, 2],
}
# A GRID is a graph too: each cell is a node, neighbours are up/down/left/right.
The one thing trees didn't need: a visited set Because graphs have cycles, you MUST track which nodes you've already seen — or you'll loop forever. Every graph traversal carries a visited set. This is the #1 difference from tree traversal.

The two traversals (memorise these templates)

TraversalToolBest for
BFS (breadth-first)a queueShortest path in an unweighted graph; exploring by distance/level.
DFS (depth-first)recursion (or a stack)Connectivity, cycle detection, paths, "fill this region".
from collections import deque
def bfs(start, graph):
    visited = {start}
    q = deque([start])
    while q:
        node = q.popleft()
        for nb in graph[node]:
            if nb not in visited:    # the crucial check
                visited.add(nb)
                q.append(nb)

def dfs(node, graph, visited):
    visited.add(node)
    for nb in graph[node]:
        if nb not in visited:
            dfs(nb, graph, visited)

1 · Number of Islands (the classic grid problem)  ＋medium

Count islands of "1" (land) in a grid of "1"/"0". Connected horizontally/vertically.

Approach — DFS flood-fill, sinking each island best
def num_islands(grid):
    rows, cols = len(grid), len(grid[0])
    count = 0
    def sink(r, c):                  # DFS: erase a whole connected island
        if (r < 0 or r >= rows or c < 0 or c >= cols
                or grid[r][c] == "0"):
            return
        grid[r][c] = "0"             # mark visited by sinking to water
        sink(r+1,c); sink(r-1,c); sink(r,c+1); sink(r,c-1)   # 4 directions
    for r in range(rows):
        for c in range(cols):
            if grid[r][c] == "1":    # found unvisited land = a new island
                count += 1
                sink(r, c)
    return count
# Time O(rows·cols)  ·  Space O(rows·cols)

The idea: the grid is a graph (each cell a node). Scan for land; each time you hit an unvisited "1", that's a new island — DFS out from it and "sink" (mark) every connected cell so you don't count it again.

2 · Flood Fill  ＋easy

Like the paint-bucket tool: repaint a connected region of same-coloured pixels to a new colour.

Approach — DFS from the start pixel best
def flood_fill(image, sr, sc, new):
    old = image[sr][sc]
    if old == new:                   # already that colour -> avoid infinite loop
        return image
    def dfs(r, c):
        if (r < 0 or r >= len(image) or c < 0 or c >= len(image[0])
                or image[r][c] != old):   # stop at edges or different colours
            return
        image[r][c] = new
        dfs(r+1,c); dfs(r-1,c); dfs(r,c+1); dfs(r,c-1)
    dfs(sr, sc)
    return image
# Time O(rows·cols)  ·  Space O(rows·cols)

Same engine as Number of Islands. Note the old == new guard — without it, painting the same colour would recurse forever (here, the repaint itself is what "marks visited").

3 · Rotting Oranges (multi-source BFS)  ＋medium

Grid: 2 = rotten, 1 = fresh, 0 = empty. Each minute, rotten oranges rot their 4-neighbours. Minutes until none fresh (or −1).

Approach — BFS from ALL rotten oranges at once best
from collections import deque
def oranges_rotting(grid):
    q = deque(all_rotten_cells)      # multi-source: every rotten orange is a start
    fresh = count_of_fresh
    minutes = 0
    while q and fresh:
        minutes += 1
        for _ in range(len(q)):      # one minute = one BFS level
            r, c = q.popleft()
            for (nr, nc) in fresh_neighbours(r, c):
                grid[nr][nc] = 2; fresh -= 1; q.append((nr, nc))
    return minutes if fresh == 0 else -1
# Time O(rows·cols)  ·  Space O(rows·cols)

Multi-source BFS: start the queue with every rotten orange, then spread level by level — each level is one minute. The "process exactly len(q) nodes per level" trick (from tree BFS) measures the minutes.

4 · Clone Graph  ＋medium

Make a deep copy of a graph (nodes + neighbour links), handling cycles.

Approach — DFS with a hash map (original → clone) best
def clone_graph(node):
    if not node: return None
    clones = {}                      # original node -> its clone
    def dfs(n):
        if n in clones:              # already cloned -> reuse (breaks cycles)
            return clones[n]
        copy = Node(n.val)
        clones[n] = copy             # record BEFORE recursing into neighbours
        for nb in n.neighbors:
            copy.neighbors.append(dfs(nb))
        return copy
    return dfs(node)
# Time O(V + E)  ·  Space O(V)

The crucial line: store the clone in the map before recursing into neighbours. That way, when a cycle leads back to a node, you return its existing clone instead of looping forever. The hash map does double duty: cycle-breaker and clone-linker.

▶ Watch it clone (and break the cycle) step by step
Talk-track"DFS with a map from each original node to its clone. I create and record a node's clone before recursing into its neighbours, so revisiting a node in a cycle just returns the existing clone. O(V + E)."
5 · Course Schedule (cycle detection)  ＋medium

Given courses and prerequisites, can you finish them all? (Possible ⇔ the prerequisite graph has no cycle.)

Approach — DFS with 3 states to detect a back-edge (cycle) best
def can_finish(num, prereqs):
    graph = build_adjacency(prereqs)
    state = [0] * num                # 0=unvisited, 1=visiting, 2=done
    def dfs(c):
        if state[c] == 1: return False   # back to a node we're STILL visiting = cycle
        if state[c] == 2: return True    # already cleared
        state[c] = 1                     # mark "in progress"
        for nxt in graph[c]:
            if not dfs(nxt): return False
        state[c] = 2                     # fully explored, no cycle through here
        return True
    return all(dfs(c) for c in range(num))
# Time O(V + E)  ·  Space O(V)

Key idea: a cycle means an impossible dependency loop. The "visiting" state (1) catches it: if DFS reaches a node that's still on the current path, you've found a cycle. This is the basis of topological sort.

▶ Watch the states change step by step
Talk-track"Courses are doable iff the prerequisite graph is acyclic. I DFS with three states; if I ever reach a node that's still 'visiting' on the current path, that's a back-edge — a cycle — so return false. O(V + E). It generalises to topological sort."
6 · Number of Connected Components  ＋medium

How many separate "clusters" in an undirected graph? n=5, edges=[[0,1],[1,2],[3,4]] → 2.

Approach — DFS from each unvisited node best
def count_components(n, edges):
    adj = build_adjacency(edges)
    visited = set()
    def dfs(node):
        visited.add(node)
        for nb in adj[node]:
            if nb not in visited:
                dfs(nb)              # mark everything reachable
    count = 0
    for i in range(n):
        if i not in visited:         # a fresh node = a new component
            count += 1
            dfs(i)
    return count
# Time O(V + E)  ·  Space O(V)

The pattern: exactly like Number of Islands but on an explicit graph. Each unvisited node launches a DFS that marks its entire cluster; every launch is one component. (Union-Find is the alternative — worth a mention in interviews.)

1 · Bubble Sort  ＋easy

Repeatedly swap adjacent out-of-order pairs; the largest "bubbles" to the end each pass.

def bubble_sort(arr):
    n = len(arr)
    for i in range(n):
        swapped = False
        for j in range(n - 1 - i):           # last i items are already sorted
            if arr[j] > arr[j+1]:
                arr[j], arr[j+1] = arr[j+1], arr[j]   # swap neighbours
                swapped = True
        if not swapped:                       # a clean pass -> already sorted, stop
            break
    return arr
# Best O(n) (early exit) · Avg/Worst O(n²) · Space O(1) · Stable
▶ Watch it bubble step by step
Talk-track"Bubble sort repeatedly swaps adjacent out-of-order elements; each pass floats the next-largest to the end. With an early-exit flag it's O(n) on already-sorted data, otherwise O(n²). Stable, O(1) space. Rarely used in practice — mostly a teaching sort."
2 · Selection Sort  ＋easy

Find the smallest remaining element and put it in the next position.

def selection_sort(arr):
    n = len(arr)
    for i in range(n):
        min_idx = i
        for j in range(i+1, n):
            if arr[j] < arr[min_idx]:
                min_idx = j                   # track the smallest in the unsorted part
        arr[i], arr[min_idx] = arr[min_idx], arr[i]   # place it at position i
    return arr
# Always O(n²) · Space O(1) · NOT stable
▶ Watch it select step by step
Talk-track"Selection sort scans the unsorted part for the minimum and swaps it to the front each round. Always O(n²) because it scans the rest every time, regardless of input. O(1) space, and it's not stable."
3 · Insertion Sort  ＋easy

Like sorting playing cards in your hand — take each new card and slide it into the right spot among the ones already sorted.

def insertion_sort(arr):
    for i in range(1, len(arr)):
        key = arr[i]
        j = i - 1
        while j >= 0 and arr[j] > key:        # shift bigger items one step right
            arr[j+1] = arr[j]
            j -= 1
        arr[j+1] = key                        # drop the key into its slot
    return arr
# Best O(n) (nearly sorted) · Avg/Worst O(n²) · Space O(1) · Stable
▶ Watch it insert step by step
Talk-track"Insertion sort builds a sorted prefix, inserting each new element by shifting larger ones right. It's great on nearly-sorted data — O(n) best case — and stable with O(1) space. It's what Timsort uses for small runs."
4 · Merge Sort (divide & conquer)  ＋medium

Split in half, sort each half (recursively), then merge the two sorted halves.

def merge_sort(arr):
    if len(arr) <= 1:                # base case: 0 or 1 element is sorted
        return arr
    mid = len(arr) // 2
    left = merge_sort(arr[:mid])     # sort the left half
    right = merge_sort(arr[mid:])    # sort the right half
    return merge(left, right)        # combine two sorted lists

def merge(a, b):
    result, i, j = [], 0, 0
    while i < len(a) and j < len(b):
        if a[i] <= b[j]: result.append(a[i]); i += 1   # take the smaller front
        else:            result.append(b[j]); j += 1
    return result + a[i:] + b[j:]    # append whatever's left
# O(n log n) ALWAYS · Space O(n) · Stable

Why O(n log n): halving gives ~log n levels; merging each level touches all n elements. Guaranteed — no bad case. The price is O(n) extra space.

▶ Watch it divide & merge step by step
Talk-track"Merge sort splits the array in half, recursively sorts each, and merges the two sorted halves by repeatedly taking the smaller front element. It's O(n log n) in all cases and stable, at the cost of O(n) extra space. Great when you need guaranteed performance or stability."
5 · Quick Sort (divide & conquer)  ＋medium

Pick a pivot, partition into "smaller" and "bigger", recursively sort each, and concatenate.

def quick_sort(arr):
    if len(arr) <= 1:
        return arr
    pivot = arr[-1]                                  # choose a pivot (here: last element)
    smaller = [x for x in arr[:-1] if x <= pivot]    # everything ≤ pivot
    bigger  = [x for x in arr[:-1] if x >  pivot]    # everything > pivot
    return quick_sort(smaller) + [pivot] + quick_sort(bigger)
# Avg O(n log n) · Worst O(n²) (bad pivots) · Space O(log n) in-place · NOT stable

The catch: if the pivot is always the smallest/largest (e.g. already-sorted input with a naive pivot), partitions are lopsided → O(n²). Real implementations pick a random or median pivot to avoid that. (This readable version isn't in-place; the classic in-place partition is O(log n) space.)

▶ Watch it partition step by step
Talk-track"Quicksort picks a pivot, partitions into smaller and bigger groups, and recurses. Average O(n log n) and usually fastest in practice with in-place partitioning, but a bad pivot gives O(n²) — so good implementations randomize the pivot. Not stable."

🗣️ Part 1 — 20 Verbal Questions

Theory questions interviewers ask out loud. Answer first, then click to check.

1 · What is Big-O / time complexity?theory
Show answer

Notation for how an algorithm's number of steps grows as the input size n grows, ignoring constants, using the worst case. e.g. O(n) means the work grows in direct proportion to n.

2 · Array vs Linked List — trade-offs?theory
Show answer

Array: contiguous memory, O(1) index access, O(n) insert/delete (shifting). Linked list: nodes joined by pointers, O(n) to reach the k-th node, but O(1) insert/delete given the node. Arrays for random access; lists for frequent insert/delete at known spots.

3 · What is a hash map and its complexity?theory
Show answer

A key→value store (Python dict) using hashing for average O(1) insert, lookup, and delete. Perfect for "have I seen this?" checks and counting frequencies.

4 · When does a hash map degrade to O(n)?theory
Show answer

When many keys collide into the same bucket (a poor hash function or adversarial input). Good hashing plus resizing keep operations O(1) on average.

5 · Stack vs Queue?theory
Show answer

Stack = LIFO (last in, first out): push/pop the top — like a plate stack or undo. Queue = FIFO (first in, first out): add at back, remove at front — like a line. Both have O(1) operations. Use a deque for a queue in Python.

6 · BFS vs DFS — what's the difference?theory
Show answer

BFS explores level by level using a queue. DFS goes as deep as possible down one path using recursion (or a stack) before backtracking.

7 · When do you use BFS vs DFS?theory
Show answer

BFS for shortest path in an unweighted graph or anything by distance/level. DFS for connectivity, cycle detection, exploring all paths, or "fill this region".

8 · What is a Binary Search Tree? Search complexity?theory
Show answer

A binary tree where for every node, the left subtree is smaller and the right subtree is larger. Search/insert/delete are O(h) — O(log n) if balanced, O(n) if degenerate. Inorder traversal yields sorted order.

9 · What does "stable" mean for a sort?theory
Show answer

Equal elements keep their original relative order. It matters when sorting by multiple keys. Merge and insertion sort are stable; selection sort and plain quicksort are not.

10 · Merge sort vs Quick sort?theory
Show answer

Merge: O(n log n) always, stable, needs O(n) extra space. Quick: average O(n log n) and in-place, but O(n²) worst case with bad pivots and not stable. Quick is usually faster in practice with a randomised pivot.

11 · Why is binary search O(log n)? Precondition?theory
Show answer

It halves the search space each step, so it takes about log₂(n) steps. Precondition: the data must be sorted (or have a monotonic yes/no condition).

12 · What is recursion? Its two parts?theory
Show answer

A function that calls itself on a smaller input. It needs a base case (a stop condition answered directly) and a recursive case (a call on a smaller input that progresses toward the base).

13 · What is the call stack / stack overflow?theory
Show answer

Each function call is pushed onto the call stack and pauses while its sub-calls run. Recursion that is too deep or missing a base case overflows the stack (Python: "maximum recursion depth exceeded").

14 · What is dynamic programming / memoization?theory
Show answer

Solving a problem by combining solutions to overlapping subproblems. Memoization caches each subproblem's answer so it is computed once (top-down), turning exponential recursion into polynomial — e.g. Fibonacci from O(2ⁿ) to O(n).

15 · What counts toward space complexity?theory
Show answer

The extra memory used as a function of n — new arrays/maps/sets you create, plus the depth of the recursion call stack. The input itself usually is not counted.

16 · What is the two-pointer technique?theory
Show answer

Use two indices moving through the data — from both ends inward, or a fast/slow pair — to avoid a nested loop. It often turns O(n²) into O(n), especially on sorted arrays.

17 · What is the sliding window technique?theory
Show answer

Maintain a window over a contiguous range; as it moves, add the entering element and remove the leaving one instead of recomputing from scratch — giving O(n) for subarray/substring problems.

18 · What is a monotonic stack?theory
Show answer

A stack kept in increasing or decreasing order. It solves "next greater/smaller element" style problems in O(n), because each element is pushed and popped at most once.

19 · How do you detect a cycle in a linked list?theory
Show answer

Floyd's tortoise & hare: a slow pointer moving 1 step and a fast pointer moving 2 steps. If they ever meet, there's a cycle; if fast reaches the end, there isn't. O(1) space.

20 · Complexity of common operations?theory
Show answer

Array index a[i]: O(1). Append to a list: O(1) amortised. Searching a value in a list: O(n). dict/set lookup: O(1) average. Sorting: O(n log n).

Step 0 — Every problem is the same shape

Almost every interview problem is really saying three things:

"Here is some data → give me an answer about it → and do it fast."

Your job has two halves: (1) the obvious slow way (the brute force), and (2) a faster way using the right tool. The entire game is matching the clues in the problem to the right tool. The interviewer is testing whether you can read those clues — nothing more magical than that.

The 3 tools beginners mix up the most

Get these three crystal clear and most of your confusion disappears.

🗝️ Hash map (Python dict / set)

Real life A phone book. You know the name (the key) and instantly get the number (the value) — you do NOT read every page. Or a coat-check: hand over the ticket, get your exact coat back in one step.

What it is, plainly: a box that stores key → value pairs and hands you the value in one step when you give the key. No scanning.

The tell — reach for it when you keep wanting to ask:

  • "Have I seen this value before?"
  • "How many times does this appear?" (counting / frequency)
  • "What was the index of this thing?"
  • "Does the partner I need exist?"
The giveaway If you are about to write a loop inside a loop just to find or count something → stop and use a hash map. It turns that inner search from O(n) into O(1), so the whole thing drops from O(n²) to O(n).

↔️ Two pointers

Real life Two friends closing a book from both ends, walking toward the middle. Or zipping a jacket — two sides meeting.

What it is, plainly: two index markers moving through the data — usually one from the left and one from the right, moving toward each other (sometimes one slow, one fast).

The tell:

  • The array is SORTED and you want a pair (or triplet) that meets a condition.
  • You compare from both ends (palindrome, reversing).
  • You must do it in place, no extra memory.

Why it works (plain): because it is sorted, moving the left pointer right makes the sum bigger, moving the right pointer left makes it smaller — so you always know which one to move. You never try every pair.

🪟 Sliding window

Real life Looking out of a moving train window — you see a fixed group of seats; as the train rolls one seat forward, one new seat enters your view and one leaves. You never re-count all of them.

What it is, plainly: a contiguous range [left…right] that grows and shrinks as you slide along, reusing work instead of recomputing.

The tell — these WORDS in the problem: "longest / shortest / maximum / minimum" combined with "contiguous", "subarray", "substring", "consecutive", or "window of size k".

Two pointers vs sliding window — the confusion, settled Both use two indices, so they feel the same. Tell them apart by the question:
• Looking for a PAIR / two things (usually on sorted data)? → two pointers (they meet in the middle).
• Looking for the best CONTIGUOUS run (subarray/substring)? → sliding window (a frame that slides, both indices move left-to-right).
Memory hook: two pointers = "meet in the middle"; sliding window = "a frame that slides."

1 · Convert "aabbccddd" → "a2b2c2d3"string

Run-Length Encoding: write each letter followed by how many times it repeats in a row.

Deciding (4 questions): Data = a string with letters grouped together. Asking = letter + run-length. Brute force = none needed, one pass is enough → O(n). Tool = a single pass with a counter: track the current letter and how many in a row; when it changes, write letter+count and reset.

def encode(s):
    if not s: return ""
    result = []
    count = 1
    for i in range(1, len(s)):       # compare each char to the one before
        if s[i] == s[i-1]:
            count += 1               # same run → keep counting
        else:
            result.append(s[i-1] + str(count))   # run ended → record it
            count = 1
    result.append(s[-1] + str(count))   # ⚠ write the LAST run (the bit everyone forgets!)
    return "".join(result)
# O(n) time, O(n) space
The bug that fails peopleThe loop only writes a group when the letter changes. After the final d nothing changes, so the last group must be written after the loop. Forget it and you get a2b2c2 — the d3 vanishes.

3 · Search for an element — fastest waysearch

First ask the magic question: "Is the array sorted?" — that one clue decides everything.

def binary_search(nums, target):   # nums must be SORTED
    lo, hi = 0, len(nums)-1
    while lo <= hi:
        mid = (lo+hi)//2
        if nums[mid] == target: return mid
        elif nums[mid] < target: lo = mid+1
        else: hi = mid-1
    return -1
# O(log n) time

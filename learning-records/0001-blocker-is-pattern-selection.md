# The real blocker is pattern-selection, not solutions

After a real DSA interview (2026-06-26) the user could not solve any problem. They
diagnosed it themselves: they don't know **which technique to use, when, or why** — they
get confused choosing between hash map / two pointers / sliding window / etc. They have
never studied DSA before.

**Why it matters:** the course was too solution-centric. The user needs the *decision
layer* taught first — "given a new problem, how do I read the clues and pick the right
tool, and why is it best?" — in basic language, with real-life examples, step by step deep.
Only once that clicks can they solve independently.

**How to apply:** Lead with concept clarity + a decision framework (signal → tool → why).
Drill pattern-recognition ("which technique fits this problem?") as its own skill. Keep
explanations plain and analogy-first. Then move to guided problem solving where we reason
problem → signal → tool → solution together. See [[MISSION.md]].
